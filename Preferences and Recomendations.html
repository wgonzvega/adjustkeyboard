Goal

To serve as a guide for all users that needs to evaluate, design, deploy or configure cloud related resources in Azure or AWS environments.

 

Introduction

The process to configure or deploy cloud resources can be at first a challenge all by itself. This document will pave a path to have a clearer picture of what type of resources are preferred or recommended over others in what cloud provider the actual deployment is recommended or preferred. This list is by no mean mandatory, there could be exceptions to these list or terms of decisions, if thats the case please contact the Enterprise Architecture Team to help you address any concern that you may have.

 

General Information

Scope

This guide applies to any user that needs to deploy or configure cloud resources. Any other resources that are not listed in this document we recommend having it discussed with the Enterprise Architecture Team.

 

Preferences and Recommendations

The Enterprise Architecture Team recommends and /or preferers the following postulates:

 

      We recommend Serverless or Managed Services on Server

      We recommend DB handled over DB on Servers

      We prefer AWS for Oracle over Oracle on Azure

      We prefer AWS for Oracle deployments when we have deployments of 16 vCores or less

      We prefer Amazon RDS for deployments of less than 16 vCores for SE2 license type

      We prefer ??? for deployments that have EE license

      We prefer Azure when we have deployments of more than 16 vCores

      We prefer Microsoft SQL for Azure over SQL Server on AWS

      We must use Fortigate and Big IPs F5 for environments with a web presence through Corenetwork

      We prefer native load balancer services over solutions like Fortigate and Big IPs F5 when services are not public

      We prefer computer services reserved for on-demand services in production environments

      We prefer on-demand services instead of reserved for development and certification environments

      We prefer the communication of private type services public communications

      We recommend deploying Infrastructure DevOps and DevSecOps as code (CI/CD) rather than manual provisioning (automation) Preference the DevSecOps model instead of dev, sec and ops operating as separate units

      We prefer oriented architectures as microservices rather than conventional architectures

      We prefer the segregation of environments instead of opting for combined environments (abound on the right side of the sentence)

      We prefer centralized monitoring to decentralized monitoring

      We prefer Azure WebApp over AWS WebApps

      We recommend deployments in multiple regions to deployments in singular regions

      We prefer resource reuse rather than duplication

      We must use the Naming Convention as a standardization of resource names

      We prefer to outsource logs to a central point rather than queries isolated by environments

      We prefer AutoScaling groups over vertical growth

      We prefer messaging service on AWS instead of Azure because of the costs of the service
